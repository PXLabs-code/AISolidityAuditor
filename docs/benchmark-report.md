# Benchmark report

## Purpose

This report tracks whether AISolidityAuditor adds useful, reproducible triage value on top of raw Slither output without overstating security guarantees.

## Current benchmark scope

The current CI-backed evaluation suite covers 30 Solidity fixtures in `examples/evaluation`.

For each fixture, CI checks:

- Slither runs against the Solidity source.
- Expected detector IDs are present.
- Expected detector severities match the manifest.
- SARIF generation succeeds.
- Assistive explanation payloads pass the same grounding validator used for AI responses.

The Glamsterdam readiness extension adds source-pattern tests for upgrade-readiness prompts. These checks are intentionally conservative and are measured separately from Slither vulnerability detectors.

## Raw Slither vs AISolidityAuditor comparison

| Review task | Raw Slither | AISolidityAuditor |
|-------------|-------------|-------------------|
| Find detector ID and severity | Available in JSON/CLI output | Normalized in findings JSON and report |
| Locate source evidence | Requires reading Slither elements | Source context is attached to each finding when available |
| Share with non-security reviewers | Requires manual formatting | Markdown report and PR comment |
| Track in GitHub code scanning | Requires SARIF conversion | SARIF generated by default |
| Understand next manual-review step | Requires detector knowledge | Assistive explanation and recommendation |

## Manual usability benchmark plan

Run the same fixture corpus with two reviewer groups:

- Group A receives raw Slither output.
- Group B receives AISolidityAuditor reports.

Measure:

- Time to identify the affected function.
- Time to state the likely impact.
- Whether the reviewer correctly identifies that manual review is still required.
- Whether the reviewer over-trusts a false-positive case.

## Optional live AI benchmark

The default CI suite does not call paid model APIs. A maintainer can run a separate benchmark with provider keys configured and record:

- AI response success rate.
- JSON schema failure rate.
- Detached explanation rejection rate.
- Manual-review language preservation.
- Per-provider latency and cost.

## Expansion backlog

Target fixture categories for future expansion:

- Reentrancy variants.
- Unchecked low-level calls.
- `tx.origin` authorization.
- Timestamp and block-number dependence.
- Delegatecall and proxy misuse.
- Selfdestruct and lifecycle hazards.
- Access-control mistakes.
- Missing events.
- Solidity version and compiler configuration issues.
- ERC20/ERC721 integration pitfalls.
- Oracle and price-feed assumptions.
- Upgradeability storage-layout hazards.
- Clean contracts and intentional false-positive cases.
